# Монитор билетов БЖД

Локальный и серверный сервис для проверки появления мест на `pass.rw.by`. Пользователь добавляет маршрут или номер поезда, дату и время окончания мониторинга. Сервис проверяет наличие мест каждые несколько минут и присылает сообщение в Telegram, как только места появляются.

## Как это работает

`pass.rw.by` закрыт анти-бот защитой: на прямой HTTP-запрос сайт отвечает JS-проверкой (страница `Verification`) и блокировкой WAF (`Запрос заблокирован`, 403). Поэтому проверка идёт через реальный браузер (Playwright + Chromium):

- настоящий браузерный fingerprint проходит WAF;
- браузер сам выполняет JS-проверку (с маскировкой `navigator.webdriver`, иначе сайт не выдаёт cookie доступа);
- наличие мест определяется по DOM (класс `w_places` у поезда), а не по тексту;
- один экземпляр браузера переиспользуется, cookies хранятся в `data/browser-profile`, между проверками выдерживается интервал со случайным джиттером — чтобы не провоцировать блокировку.

Перед первым запуском нужно один раз установить браузер Playwright:

```bash
npx playwright install chromium
```

## Локальный запуск

1. Установить зависимости:

```bash
npm install
```

2. Создать `.env` по примеру `.env.example`:

```bash
PORT=4174
CHECK_INTERVAL_MS=180000
CHECK_JITTER_MS=45000
MAX_CHECKS_PER_TASK=0
TELEGRAM_MIN_INTERVAL_MS=60000
TELEGRAM_BOT_TOKEN=токен_бота
TELEGRAM_CHAT_ID=chat_id
```

`MAX_CHECKS_PER_TASK=0` — мониторить без лимита (до появления мест или до даты `Мониторить до`). Поставьте число, если нужно ограничить количество проверок.

3. Запустить сервис:

```bash
npm run dev
```

Панель откроется на `http://localhost:5173`, API работает на `http://localhost:4174`.

## Продакшн-запуск

Сервис умеет работать одним Node-процессом: после сборки сервер отдает и API, и веб-панель.

```bash
npm ci
npm run build
npm start
```

После этого приложение доступно на `http://localhost:4174`.

## Деплой не с домашнего компьютера

GitHub Pages не подходит: там нет постоянного сервера и фонового таймера. Нужен хостинг, где можно запустить Node + headless-Chromium: VPS, Docker-хостинг, Fly.io, Railway. Из-за Playwright (браузер) сборка тяжелее обычного Node-приложения — на бесплатных тарифах с маленькой памятью браузер может не запуститься.

Для Render есть `render.yaml`, но в build нужно доустановить браузер:

1. Загрузите проект в GitHub-репозиторий.
2. В Render создайте `New` -> `Blueprint` и выберите репозиторий.
3. В переменных окружения задайте `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `CHECK_INTERVAL_MS=180000`, `MAX_CHECKS_PER_TASK=0`, `TELEGRAM_MIN_INTERVAL_MS=60000`.
4. После деплоя откройте публичный URL Render. На нем будет та же панель.

Самый предсказуемый вариант — Docker (образ Playwright уже содержит Chromium):

```bash
docker build -t rw-ticket-watch .
docker run -p 4174:4174 --env-file .env rw-ticket-watch
```

## Telegram

Чтобы узнать `TELEGRAM_CHAT_ID`, напишите боту любое сообщение, затем откройте:

```text
https://api.telegram.org/bot<токен>/getUpdates
```

В ответе нужен `message.chat.id`. После настройки нажмите в панели кнопку `Проверить Telegram`.

## Проверка билетов

По умолчанию сервис строит ссылку поиска на `pass.rw.by` из маршрута, даты и номера поезда. Если сайт изменит параметры или для конкретного маршрута нужна точная страница результатов, откройте поиск вручную в браузере и вставьте готовую ссылку в поле `Точная ссылка поиска`.

Данные задач хранятся локально в `data/watch-tasks.json`. Файл не коммитится, чтобы не сохранять личные маршруты.
